## Executive decision

Sangam 0.1.0 is released. The seven feature phases and the cross-cutting release
hardening are merged into protected `main`, the `v0.1.0` tag points to the merge
commit above, and the public release contains a signed multi-platform application
image plus attested Python artifacts and checksums.

The release is suitable for a technically capable operator to deploy privately by
immutable image digest. It is not a claim that a particular production instance has
passed Cloudflare Access, ChatKit domain registration, provider, monitoring, or
off-host restore acceptance. Those checks require the deployment's credentials,
hostnames, identities, storage, and failure domains and remain explicit operator
gates.

This report closes the management question raised by the original
[readiness audit](./0.1_READINESS_AUDIT.md). That audit is retained as the evidence
baseline at pre-hardening revision `33610e5`; this document records the implemented
result and remaining work.

## Management and release controls

The repository now has the minimum credible release-management system for a 0.1
self-hosted product:

- `main` requires an up-to-date pull request, all three CI contexts, and resolved
  conversations; direct administrator pushes, force-pushes, and deletion are
  blocked;
- CI verifies source, UI, documentation, configuration, clean package installs,
  Compose, the production image, and the container security policy;
- the release workflow re-runs the gates before it can publish, then builds amd64 and
  arm64 images, creates SBOM and provenance, signs the immutable digest, attests
  artifacts, and creates the GitHub Release;
- base images and GitHub Actions are digest or commit pinned;
- Dependabot, CODEOWNERS, a security policy, changelog, release checklist, and
  upgrade/rollback runbook are present; and
- the supported deployment path is the signed container by digest, with a separate
  development Compose file and fail-closed production configuration.

The project is still maintained by one person, so branch protection requires the PR
and automated evidence but does not require an approval that the author cannot
provide to their own pull request. Add an independent required review when a second
maintainer is available.

## Release-blocker closure

Every P0 item from the readiness audit was addressed before the tag:

1. **Document materialization consistency.** Per-document coordination now covers
   the complete database, filesystem materialization, and search sequence. A backup
   barrier is fair with concurrent mutations.
2. **Backup generation consistency.** Backup creation captures a consistent paired
   database/workspace generation. Verification cross-checks database heads,
   revision hashes, archived bytes, duplicate members, unsafe paths, and PDFs.
3. **Audit-event identity.** Events have distinct primary keys; the operation ID is
   retained as non-unique request/turn correlation.
4. **Chat administration.** Agent credentials cannot mutate or refresh global chat
   model settings.
5. **Release pipeline.** Protected CI and the release workflow are live on `main`
   and proved by the published release.
6. **Distribution contract.** The wheel and source distribution build and install
   cleanly with all 14 migrations; the container is the complete application.
7. **Configuration parity.** A checked inventory keeps 44 settings aligned across
   code, environment examples, development Compose, production Compose, and docs.
8. **Production fail-closed behavior.** Production mode rejects known development
   secrets, insecure URLs, and incompatible authentication/origin configuration.
9. **Browser draft truthfulness.** Storage initialization, write, and delete
   failures produce persistent recovery UI with retry and copy/download paths.

## Product and engineering state

### Document core and storage

SQLite remains canonical for identity, immutable revisions, metadata, trust,
publications, annotations, imports, and chat state. Workspace files are controlled
materializations. Optimistic revision checks, idempotency, actor attribution,
reconciliation, trash/restore, search, and paired backups use the normal service
path. Startup reconciliation can now close stale conflicts and repair safe drift.

The main remaining lifecycle gaps are folder-level move/rename/delete, PDF
move/trash and controlled purge, batch workspace onboarding, local asset upload and
resolution, and a portable versioned export distinct from a Sangam restore.

### Backend and API reliability

Readiness now covers database/schema access, writable roots, startup reconciliation,
pending materializations, and backup freshness. Proposal reservations recover from
failure; PDF upload and range delivery are streamed and bounded; extraction work is
atomically claimed and recoverable; shutdown is bounded; misleading Karakeep
idempotency behavior was removed; and migration filenames, numbers, checksums, and
unknown-newer schemas are governed.

Collections that can grow without bound still need cursor pagination, especially
revision metadata/bodies, annotations, PDF pages, proposals, publications, and
imports. Validation and unexpected failures should converge on one problem schema
with an operation ID. Multi-process write coordination is outside the 0.1 single
application-process deployment contract and must be designed before horizontal
replicas are supported.

### Security, privacy, and trust

Agent tokens are one-time secrets stored as hashes, default to short expiry and
least privilege, separate read/search/write prefixes, and warn before dangerous
capabilities. Path syntax is canonicalized before authorization; physical
workspace/symlink containment is checked after authorization. Trusted HTML preview,
published HTML, and the application remain separate trust zones. Chat publication
and other external side effects require an explicit browser confirmation, while
existing-document edits remain revision-pinned proposals.

The release does not eliminate operator responsibility. Secrets must be supplied by
the deployment environment, logs and monitoring must avoid document bodies and
tokens, and the production Access/Tunnel/preview-host policy must be exercised with
both an allowed and denied identity.

### UI, UX, accessibility, and performance

Conflict recovery shows base, server, and local states with copy, download, rebase,
retry, and confirmed discard. Citations open the exact revision, PDF page, and
annotation and expose staleness. Draft failure recovery, token scope previews,
destructive confirmations, keyboard tabs and splitters, live status, contrast,
focus trapping, Escape behavior, narrow-screen modal navigation, and route-close
behavior were implemented.

The editor and preview surfaces own their overflow at desktop and narrow widths.
ChatKit is deferred until chat is activated; inactive inspector queries are gated;
the document list is cached once; and heavyweight routes are lazy-loaded.

The next quality step is to put full Playwright workflows and automated
accessibility checks in CI, add screen-reader acceptance, define JavaScript and
interaction budgets, continue breaking up large editor/PDF/diff/language chunks,
and standardize loading/error/retry states across every route.

### Publishing, PDF, Karakeep, and chat

- Publishing supports stable private, public, and unlisted routes; trusted preview
  remains isolated and published HTML remains sanitized.
- PDFs are immutable Documents with streamed byte ranges, extracted page search,
  exact citations, actor-attributed annotations, and replacement relationships.
- Karakeep remains the archive of record; selective imports are normal editable
  Documents and refreshes cannot overwrite human corrections.
- Chat has durable threads, streaming, model selection, revision-pinned citations,
  explicit proposal review, and explicit confirmation for publication side effects.

The state-of-the-art roadmap remains intentionally focused: better search filters
and backlinks before vectors or a graph; stable Markdown anchors, outlines, assets,
and portable citations; annotation-to-note synthesis and research export; and MCP
interoperability only after the current API/capability model remains stable in use.

## Boundaries not verified by this release

The following are not hidden failures; they require a real operator environment and
must be recorded before describing a deployment as production-accepted:

- a pre-upgrade paired backup from real data, encrypted replication to a separate
  failure domain, checksum verification there, and clean-host restore drill;
- migration rehearsal against a restored copy of the deployment database and
  workspace, with the previous digest and backup ID recorded;
- Cloudflare Access and Tunnel allow/deny evidence for the application and ChatKit
  API, including streaming behavior;
- isolated trusted-preview hostname, CSP, parent origin, and rejection of the main
  application hostname;
- external public, unlisted, and private publication checks;
- production ChatKit domain registration and a real OpenRouter streaming turn;
- real Karakeep connectivity and credential rotation;
- monitoring and alert delivery for readiness, backup age/failure, disk capacity,
  process restarts, job backlog, integration failures, and origin reachability; and
- an operator-observed desktop and narrow browser smoke on the deployed digest.

Use the [release checklist](./operations/RELEASE_CHECKLIST.md) and
[upgrade/rollback runbook](./operations/UPGRADES_AND_ROLLBACK.md) to collect this
evidence. A deployment should be pinned to the immutable digest in this report, not
to `0.1.0`, `0.1`, or `latest`.

## Recommended 0.1.x sequence

1. Deploy the released digest privately and complete every production acceptance
   boundary above.
2. Add off-host encrypted backup replication, alerting, and a scheduled restore
   drill with truthful copy/verification age.
3. Add CI browser workflows and accessibility automation for the critical daily-use
   paths.
4. Add collection pagination, uniform error envelopes, and operator metrics before
   supporting materially larger workspaces.
5. Complete ordinary workspace lifecycle operations and a portable export before
   broadening integrations.
6. Cut only focused patch releases from protected `main`; do not move the `v0.1.0`
   tag or mutable deployment references.

## Final assessment

The implementation progressed from a feature-complete alpha with reproduced data,
authorization, backup, draft, and release defects to a published, signed, attested,
multi-platform 0.1.0 self-hosted beta. The release artifact and its core workflows
are verified. The remaining distinction is between **released software**, which is
now true, and **a production deployment accepted in its real environment**, which
must be proven by the operator-specific gates above.
