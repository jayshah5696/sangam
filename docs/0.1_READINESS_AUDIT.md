# Sangam 0.1 readiness and state-of-the-art gap audit

> Audit date: 2026-07-22
>
> Audited revision: `33610e5` (`main`, merge of Phase 7)
>
> Scope: release management, deployment, backend/data integrity, security, UI/UX,
> accessibility, testing, operations, and product completeness
>
> Recommendation: **do not tag `v0.1.0` yet; treat the current tree as an alpha**

## Executive answer

Sangam has implemented the seven vertical slices, and the architectural spine is
real rather than aspirational: stable Document identity, SQLite-backed immutable
revisions, optimistic concurrency, ordinary workspace files, scoped agent access,
publishing, PDF research, Karakeep import, and grounded chat all exist.

However, **Sangam has not been released as version 0.1**. The number `0.1.0` appears
in the Python package, frontend package, module, OpenAPI application, and health
response, but current `main` has no version tag, GitHub Release, published image,
working release workflow, or reproducible installable artifact. The documented
Compose path also cannot configure Phase 7.

More importantly, this audit reproduced four release-blocking correctness and
authorization defects:

1. Concurrent text updates can leave an old workspace file while SQLite says the
   newest revision is clean.
2. A backup can be internally inconsistent and still pass verification.
3. A second audited agent/chat tool call in one request can crash on a duplicate
   activity primary key.
4. Any authenticated agent token can change global chat model settings.

The browser audit also found that IndexedDB draft-write failures are silently
discarded while the UI promises that offline changes remain safe in the browser.

The right management posture is:

- **Today:** source-level alpha; excellent feature prototype, unsafe release.
- **After P0 below:** `0.1.0-rc.1` deployed privately by immutable image digest.
- **After an upgrade/restore drill and production access gates:** tag `v0.1.0`.

## What is already strong

- The backend consistently treats SQLite as canonical for identity, revisions,
  metadata, trust, publications, annotations, imports, and chat state.
- Text mutations generally flow through `DocumentService`; access control is
  concentrated in `WorkspaceAccessService` and `AuthorizationPolicy`.
- Paths are canonicalized before authorization and checked for physical workspace
  containment only after capability checks.
- Agent credentials are one-time secrets stored as hashes, support expiry and
  rotation, and carry deny-by-default capability/path scopes.
- HTML has separate safe, trusted-preview, and published rendering zones.
- PDF bytes are immutable, annotations are separately versioned, and replacements
  receive new Document IDs.
- Karakeep remains the archive of record; imports preserve provenance and do not
  overwrite corrected working copies on refresh.
- Chat edits to existing documents use revision-pinned proposals and the normal
  attributed mutation path.
- Migrations are ordered and transactionally applied; backup artifacts use SQLite's
  backup API, checksums, integrity checks, safe archive validation, staging, and
  atomic promotion.
- Local source verification is healthy: 102 backend tests and 40 frontend tests
  pass, as do Python/frontend formatting, lint, TypeScript build, UI-token lint,
  Markdown links/style, and Mermaid parsing.

This foundation should be preserved. The next work is horizontal hardening, not an
eighth feature phase.

## P0: blockers before any `0.1.0` release candidate

### P0.1 Serialize revision materialization

**Problem.** The database revision commits before the workspace file write. There
is no per-Document serialization around the complete commit/materialize/index
sequence (`src/sangam/service.py:356-364`, `497-515`, `871-893`).

The audit deterministically delayed the first write while allowing a later revision
to commit and finish. The first writer then overwrote the newer file; its conditional
database update affected zero rows, but the later writer had already marked the
Document clean:

```text
SQLite head/content: B
workspace file:      A
state:               clean
```

This violates the central source-of-truth/materialization invariant and is reachable
in the current single-process deployment because synchronous FastAPI handlers run in
parallel threads.

**Required fix.** Serialize the entire mutation-to-materialization sequence per
Document, or introduce a durable generation/lease protocol that rechecks the head
before and after atomic rename and repairs a superseded write. Add delayed,
out-of-order create/update/restore/move tests through the HTTP layer.

### P0.2 Make backups generation-consistent

**Problem.** `BackupManager` copies SQLite and then separately archives the live
workspace without quiescing mutations (`src/sangam/backup.py:107-120`). Verification
validates each artifact but never compares archived files to the database head hashes
(`src/sangam/backup.py:162-202`).

The audit mutated a Document between those phases and produced a backup that returned
`valid=True` while containing database content A and workspace content B. For text,
reconciliation may repair the file from SQLite; for a PDF, SQLite cannot reconstruct
the original bytes.

**Required fix.** Hold a shared mutation barrier while capturing both artifacts, or
snapshot a durable generation and validate against it. Verification must compare
every materialized database head/file hash to the corresponding archived member.
Add mutation-during-backup tests for text, moves/deletes, and PDFs.

### P0.3 Separate audit event identity from request correlation

**Problem.** `operation_events.operation_id` is the primary key
(`src/sangam/migrations/005_agent_collaboration.sql:37-49`). Every workspace action
records a row using `Principal.operation_id` (`src/sangam/activity.py:41-64`), while
all tools executed by one ChatKit HTTP request share that value. The second audited
operation raises:

```text
sqlite3.IntegrityError: UNIQUE constraint failed: operation_events.operation_id
```

This breaks normal multi-tool agent turns and loses the intended audit trail.

**Required fix.** Give every event its own `event_id`; retain a non-unique, indexed
`operation_id` as the request/turn correlation ID. Add multi-tool human and agent
tests that record accepted, denied, conflicted, and failed actions under one request.

### P0.4 Protect global chat configuration as administration

**Problem.** `GET /chat/models`, `PUT /chat/models`, and
`POST /chat/models/refresh` use the ordinary authenticated principal dependency
instead of the administrator dependency (`src/sangam/api_chat.py:33-50`). The audit
confirmed that a read-scoped agent can disable or change the global OpenRouter model
configuration.

**Required fix.** Require the trusted human administrator for mutations and catalog
refresh. Decide explicitly whether agents need read access to the catalog. Add an
authorization matrix test covering every global settings and maintenance endpoint.

### P0.5 Land a real release pipeline on `main`

**Problem.** Current `main` contains no `.github` workflow, changelog, release
checklist, or release script. There are no Git tags, GitHub Releases, or GHCR package.
PR #16 appears merged, but it was merged into the stale `pr-13` branch, not `main`,
and predates Phases 6 and 7.

**Required fix.** Rebase the useful CI/release work from PR #16 onto current `main`
as a new reviewed change. Require the full source, docs, artifact-install, migration,
container, and vulnerability gates before publishing. Protect `main` and make those
checks required.

The donor pipeline must not be copied unchanged: it omits Phase 7 configuration, does
not run the full Docker smoke, scans after push without blocking high/critical
findings, and does not create a complete GitHub Release.

### P0.6 Fix or narrow the distribution contract

**Problem.** `uv build` fails because the package directory already contains the SQL
migrations while `pyproject.toml:35-39` force-includes the same files. Hatchling
reports a duplicate wheel path for `sangam/migrations/001_initial.sql`.

Even after that fix, the wheel does not include the built SPA, so it is not a complete
Sangam application artifact.

**Required decision.** Prefer a **container-first 0.1 application release**. Either:

- ship a separately tested CLI/backend wheel and document that it does not contain
  the browser application; or
- include/build the SPA intentionally and test the installed wheel as a complete app.

The release gate must build the sdist and wheel, install both into clean environments,
run the CLI, start the API, and apply packaged migrations.

### P0.7 Make the documented Compose deployment configure Phase 7

**Problem.** The README instructs Compose users to put OpenRouter and ChatKit values
in `.env` (`README.md:89-105`), but `compose.yaml:12-30` does not forward any of them
to the container. It also omits advertised backup controls. The Compose image is
still named `sangam:phase6`, while the `justfile` and smoke script use `phase7`.

**Required fix.** Maintain one settings inventory that is checked against
`Settings`, `.env.example`, Compose, production Compose, and operations docs. Test
both the safe unconfigured chat path and a configured provider path. Use versioned
release image names, not phase labels.

### P0.8 Add production-mode fail-closed configuration validation

**Problem.** The default/Compose preview HMAC secret is public and fixed
(`src/sangam/config.py:72-74`, `compose.yaml:22`). `single_user` deliberately treats
all local requests as the administrator and even treats private publications as
administrator-readable (`src/sangam/security.py:477-494`, `520-536`). This is only
safe behind the documented loopback boundary. The application does not have a
production profile that rejects development secrets, `local-dev` ChatKit keys,
insecure public/preview URLs, or incompatible auth/hostname combinations.

**Required fix.** Keep ergonomic local defaults, but add an explicit production
mode that refuses to start unless authentication, allowed identity, preview secret,
origins/hosts, HTTPS public URLs, and ChatKit registration are safe. Do not rely on a
warning buried in an operations document.

### P0.9 Make browser draft durability truthful

**Problem.** IndexedDB draft `set`/`delete` failures and initialization failures are
discarded rather than surfaced (`frontend/src/browserState/draftStorage.ts:17-55`,
`frontend/src/documentSessions.tsx:129-140`, `281-298`). Quota exhaustion, private
mode, browser policy, or storage corruption can therefore leave the visible promise
“Your changes remain in this browser” while no durable draft exists.

**Required fix.** Add a distinct `draft-storage-failed` state, a persistent warning,
retry, and copy/download/export action. Test quota/write/open/delete failures and do
not claim offline durability until persistence succeeds.

## P1: complete before calling 0.1 dependable for unattended personal use

### Release, upgrade, and recovery

1. **Run a pre-upgrade backup before migrations.** Application construction applies
   migrations before the backup service starts (`src/sangam/application.py:49-54`,
   `src/sangam/api.py:89-107`). The release procedure must quiesce writes, create and
   verify a paired backup, and test the migration against a restored copy first.
2. **Declare schema compatibility.** An older binary currently does not reject a
   database containing unknown newer migrations. Record migration checksums, reject
   duplicate numbers, test a populated `009 -> 013` upgrade, and state that old
   migrations are immutable.
3. **Define rollback.** For 0.1, the safe policy can be simple: pin deployments by
   image digest, and any rollback after migrations restores database and workspace
   together from the pre-upgrade set. Address stale SQLite WAL/SHM sidecars.
4. **Move backups off-host.** The current backup directory is a sibling on the same
   host volume. Replicate verified, encrypted sets to a different failure domain;
   alert on age/copy/verification failure and rehearse a clean-host restore.
5. **Make backup status truthful.** A historic `verified_at` is not permanent proof.
   Show its age, periodically reverify, and distinguish local snapshot health from
   successful off-host replication.

### Backend and API reliability

1. **Add liveness/readiness/degraded status.** `/health` always returns `ok`, even
   after a caught startup materialization failure (`src/sangam/api.py:89-95`,
   `267-269`). Readiness should cover schema/database access, writable roots, free
   space, reconciliation, pending materializations, backup age, and job backlog.
2. **Resolve stale reconciliation conflicts.** A new scan only adds conflicts; it does
   not close a persisted conflict when the disk state becomes canonical again
   (`src/sangam/reconciliation.py:191-205`, `336-354`).
3. **Bound PDF resource use.** PDF import buffers the entire request before applying
   the configured limit (`src/sangam/api_pdf.py:61-83`); every range response reads
   the entire PDF (`86-116`). Stream with a pre-buffer byte cap, move blocking work
   off the async loop, bound parsing CPU/page/time, and atomically claim imports and
   extraction jobs.
4. **Paginate durable collections.** History returns every full revision; PDF pages,
   annotations, chat proposals, publications, and imports can be unbounded. Return
   cursor/next metadata and separate revision metadata from full bodies.
5. **Repair proposal failure states.** Non-conflict apply failures can leave a
   proposal pending with a permanently reserved idempotency key. Add recoverable
   failure states or leases.
6. **Honor Karakeep idempotency headers.** Import/refresh endpoints require the
   header but discard it. Either enforce it in the service or remove the misleading
   contract.
7. **Bound shutdown.** Startup PDF extraction tasks are awaited indefinitely on
   shutdown. Cancel or bound them and recover from durable pending state on restart.
8. **Standardize errors.** Service errors use Sangam's envelope, while FastAPI
   validation and unexpected errors do not. Return a uniform problem schema and
   operation ID.

### Trust and user-control UX

1. **Build a real conflict recovery surface.** The UI says the local draft remains,
   but its only action, “Reload current revision,” replaces the editor content
   (`frontend/src/components/document/DocumentWorkspace.tsx:166-170`,
   `frontend/src/documentSessions.tsx:160-188`). Show base, server head, and local
   draft; provide copy/download, merge/rebase/retry, and confirmed discard.
2. **Open exact cited evidence.** Chat citations contain a revision and PDF page or
   annotation, but the client deep link navigates only to the current Document
   (`frontend/src/components/ChatPanel.tsx:116-121`). Open the exact revision/page/
   annotation and show whether the current head has changed.
3. **Make agent-token defaults least-privileged.** The settings UI defaults to seven
   capabilities and forces read/search to workspace-wide scope
   (`frontend/src/components/AgentAccessSettings.tsx:20-34`). Offer read-only and
   `/agents/**` writer presets, separate read/search/write prefixes, effective-scope
   preview, short default expiry, and warnings for delete/restore/publish.
4. **Require confirmation for public side effects from chat.** Existing-document edits
   use proposals, but tool-driven creation and publication can occur directly. A
   narrow confirmation card for publish/unpublish and other external effects is
   appropriate without introducing a general approval queue.
5. **Add first-run deployment status.** Explain whether the instance is local-only or
   authenticated, whether backups are off-host, and whether Karakeep, chat,
   publication, and trusted preview are actually configured.

### Workspace lifecycle and portability

1. **Reconcile the content contract.** The vision includes image Documents, while
   code supports only Markdown, HTML, and PDF (`docs/VISION.md:77-84`,
   `src/sangam/schemas.py:17-23`). For 0.1, either implement minimal image/asset
   import/view/link/export or explicitly defer it and correct the completion claim.
2. **Resolve local assets in author previews.** `MarkdownPreview` accepts an asset
   resolver, but `DocumentWorkspace` does not provide one. Add safe upload,
   resolution, and broken-asset diagnostics.
3. **Complete ordinary lifecycle operations.** Folders cannot be renamed/moved/
   deleted as units; PDFs cannot be moved or trashed; there is no controlled purge or
   storage reclamation. Immutability should prevent in-place byte replacement, not
   prevent organization and deletion.
4. **Add batch workspace onboarding.** Preview supported files, collisions, duplicate
   hashes, identities, and outcomes; import with progress/retry and a final report.
5. **Provide a portable export, not only a Sangam restore.** Export current files,
   stable-ID/path manifest, metadata/provenance, annotations, chats/citations/
   proposals, publications, and optionally revision history in a versioned format.

### UI/UX, accessibility, and performance

1. **Adopt WCAG 2.2 AA as the release target.** Add automated axe-style checks plus
   keyboard and screen-reader smoke tests for the file tree, command palette,
   resizable panels, dialogs/menus, editor, PDF annotations, chat, and settings.
   Test focus return, focus visibility, labels/descriptions, status announcements,
   non-drag alternatives, zoom/reflow, reduced motion, contrast, and target size.
   The browser audit confirmed specific gaps:
   - sidebar, inspector, and editor tabs do not implement arrow/Home/End roving focus,
     `aria-controls`, or tabpanels;
   - resize handles are pointer-only and expose no range values;
   - save/offline/error state changes have no live status/alert announcement; and
   - River, Parchment, and Cobalt muted text measured below 4.5:1 in normal-text use.
2. **Test real workflows, not only components.** Add Playwright/browser tests for
   create/edit/autosave/offline/conflict/reconnect, revision restore, import,
   publication, PDF annotation, token issue/revoke, and chat proposal apply.
3. **Keep UI state explicit.** Standardize loading skeletons, empty states, retry,
   validation, destructive confirmation, and success announcements across routes.
4. **Improve information architecture.** Keep Files, Search, Recent, and Published
   primary; move activity, reconciliation, backups, credentials, integrations, and
   model settings under a clear Advanced/Operations area. Keep document-specific
   Properties, Links, History, and Chat contextual.
5. **Set performance budgets.** The production build succeeds but warns about chunks
   over 700 kB; editor, PDF, diff, Mermaid/language, and base chunks are large. Track
   initial JS, route JS, interaction latency, and memory on representative libraries.
   Lazy-load only the syntax/diagram support a document needs.
   The browser audit also found eager history/tags/publication/chat requests while
   Properties was active, duplicate document-list requests, and ChatKit loaded on the
   empty home page. Gate queries by active surface and serve hashed assets compressed
   with immutable cache headers.
6. **Verify desktop and narrow layouts before release screenshots.** Source-level UI
   token lint passes, but a release checklist must include rendered computed styles,
   keyboard behavior, overflow, zoom, and touch/narrow operation.
   At 390x844 the core stack avoided horizontal overflow, but the desktop-open sidebar
   became a 330 px overlay without backdrop, focus containment, Escape dismissal,
   modal semantics, or navigation auto-close. Add a proper narrow-screen drawer and
   a comfortable touch-density mode.
7. **Make async failures consistent.** Search, Settings, Trash, Karakeep, PDF render,
   reconciliation, and other routes vary widely in loading/error/retry behavior.
   Reuse the documented skeleton, empty, error, retry, confirmation, and status
   anatomy; debounce/cancel search and honor Karakeep pagination.
8. **Use honest retry behavior.** Non-conflict save failures currently schedule an
   800 ms retry loop while the UI says to edit again. Use bounded exponential backoff,
   explicit Retry, and actionable error detail.

### Operations and observability

Add structured request logs and a small operator dashboard for request latency/errors,
SQLite busy time, database/workspace/backup sizes, free disk, pending/conflicted
materializations, unresolved reconciliation, extraction/import backlog and age,
backup verification/off-host copy age, integration failures, and OpenRouter timeout/
cost. Avoid logging agent secrets, preview grants, unlisted tokens, provider keys, or
document bodies.

## P2: state-of-the-art product roadmap after 0.1

### 0.2 — connected knowledge and better retrieval

Prioritize useful connected-knowledge primitives before a graph visualization:

1. Expose existing tag/category/actor filters in the main search UI.
2. Add content type, path, source, date, publication, and integration filters.
3. Add query operators, phrases/exclusions, recent and saved searches.
4. Index internal links and report incoming backlinks, outgoing links, broken links,
   and unlinked mentions.
5. Add heading/block anchors and local graph views only after the link index is
   accurate.

This matches the useful baseline documented by Obsidian's official
[Search](https://help.obsidian.md/plugins/search) and
[Backlinks](https://help.obsidian.md/plugins/backlinks) guides without copying its
plugin-platform complexity.

### 0.2 — complete Markdown and navigation

- Task lists, footnotes, stable heading anchors, and deliberate syntax highlighting.
- Clickable outline with active-heading tracking.
- Link/asset diagnostics and safe asset upload.
- Copy stable links to a Document, exact revision, heading, PDF page, or annotation.

Do not turn Sangam into a rich-text editor or IDE.

### 0.3 — research synthesis and interoperability

- Multi-select PDF annotations and create a Markdown note from them.
- Copy quotes with page/stable-source citations.
- Export annotations as JSON and Markdown; consider annotated-PDF export later.
- Capture DOI/ISBN metadata and support BibTeX/RIS or a light Zotero bridge.
- Page thumbnails, PDF outline, fit-width/page controls, and navigation history.
- Reproducible “source packets” combining source provenance, PDF annotations,
  working notes, and revision-pinned citations.

Zotero's official [PDF reader and note editor](https://www.zotero.org/support/pdf_reader)
is a useful reference for annotations-to-notes with return links. Its
[export guidance](https://www.zotero.org/support/kb/exporting) also reinforces the
important distinction between portable export and backup.

### 0.3 — agent interoperability and explainable control

- Expose read-only resources and authorized tools over MCP after the existing API and
  capability model stabilize.
- Show exactly which documents/revisions/selections enter a chat turn.
- Add citation freshness and “source changed since answer” indicators.
- Add per-tool confirmations for externally visible or destructive side effects.
- Keep model/provider/profile configuration explicit and user-selectable.

MCP is an interoperability layer, not a reason to add a second write path or an
agent scheduler.

## Scope to continue rejecting

- Multi-user real-time collaboration and live cursors.
- A general agent scheduler or multi-agent workflow builder.
- Vector infrastructure before FTS, filters, links, and citation UX are excellent.
- A live filesystem watcher or generic bidirectional sync.
- Replacing Karakeep or Zotero.
- Full CMS/static-site-generation features.
- A plugin marketplace before core contracts stabilize.
- WYSIWYG editing, IDE-grade language tooling, and native mobile apps.
- A global graph before an accurate backlink/link-health index exists.

These exclusions protect Sangam's deliberately small complexity budget.

## Step-by-step delivery plan

### Stage 1 — correctness freeze

1. Stop feature work.
2. Fix the materialization race, backup consistency, audit-event identity, and chat
   settings authorization.
3. Make browser draft persistence failures explicit and recoverable.
4. Add deterministic regression tests for all five issues.
5. Fix reconciliation self-healing, proposal failure recovery, and PDF job claims.
6. Run the complete source, docs, and minimum-Python gates.

**Exit:** no known invariant or authorization defects; all regression tests pass.

### Stage 2 — define and build the release

1. Declare the 0.1 supported content, artifact, platform, auth, and rollback contract.
2. Fix wheel packaging or explicitly declare the wheel CLI/backend-only.
3. Unify versions and align every environment variable across code/Compose/docs.
4. Land protected CI and release workflows on `main`.
5. Build multi-architecture release-candidate images with pinned base digests, SBOM,
   provenance, signature, and a blocking vulnerability policy.
6. Run the full Docker smoke against the exact candidate digest, including configured
   chat rather than only the safe-unconfigured path.

**Exit:** immutable `0.1.0-rc.1` artifact reproducible from a known commit.

### Stage 3 — deploy and recover the candidate

1. Quiesce writes; make and verify a paired pre-upgrade backup.
2. Replicate it off-host and restore it on a clean target.
3. Test migrations against that restored copy.
4. Deploy the candidate by digest.
5. Verify representative Markdown, HTML, PDF, publication, Karakeep, agent, backup,
   and chat workflows.
6. Run and record the Phase 4 and Phase 7 Cloudflare/ChatKit manual gates.
7. Exercise rollback to the previous digest plus paired pre-upgrade restore.

**Exit:** production access, upgrade, recovery, and rollback evidence is recorded.

### Stage 4 — tag `v0.1.0`

1. Freeze the commit and generate release notes/changelog.
2. Confirm runtime, frontend, API, image labels, and Git tag all agree.
3. Publish the GitHub Release and signed image digest.
4. Deploy the exact released digest, not a moving tag.
5. Preserve the previous digest and pre-upgrade backup through the rollback window.
